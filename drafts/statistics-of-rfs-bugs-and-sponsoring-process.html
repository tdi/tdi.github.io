statistics of RFS bugs and sponsoring process

For some days I have been working on some statistics of the sponsoring process in Debian. I find this to be one of the most important things that Debian has to attract and enable new contributions. It is important to know how this process works, whether we need more sponsors, how effective is the sponsoring and what are the timings connected to it.

How is it done ?

I have used Debbugs SOAP interface to get all the bugs that are filed agains sponsorship-requests pseudo package. SOAP gives a little bit of overhead because I need to download a complete list of bugs for the sponsorship-requests package, and then process them. The same information can be easily extracted from the UDD database in the future, it will be faster because SQL works faster when working with date ranges than python obviously.

The most problematic part was getting the "real done date" of a particular bug, and frankly most of my time I have spent on writing a rather dirty and complicated script. The script gets a log for a particular bug number and returns a "real done date".

What I measured ?

RFSs is a queue and in every queue we are interested in a mean time to be processed. In this case I call the metric MTTGS (mean time to get sponsored). This is a metric that gives the overall performance insight in RFS queue. Time to get sponsored (TTGS) is a number of days between the date of filing an RFS bug and the date of closing it (bug was sponsored), whereas mean time to get sponsored is calculated as a sum of TTGSs of all bugs divided by number of bugs (in a given period of time).

Besided MTTGS I have also measures typical bug related metrics:

  • number of bugs closed in a given day
  • number of bugs opened in a given day
  • number of bugs with status open in a given day
  • number of bugs with status closed in a given day

Some graphs

Below is a plot of mttgs vs. time (click for a larger one).

As you can see, the trend is roughly exponential and MTTGS tends to settle around 60 days at the end of the year 2013. This does not mean that your package will wait 60 days on average, I remind that this is a global MTTGS, so even if the MTTGS of last month was very low, the global MTTGS would decrease just slightly. It gives, however, a good glance in performance of the process. Even that more packages are files for sponsoring (see next graphs) now, than in the beginning of the epoch, the sponsoring rate is high enough to flatten global MTTGS, and with time maybe decrease it.

The image below (click for a larger one) shows how many bugs each day stay with status open or closed. For closed we have an almost linear function, so each day more or less the same amount of bugs are closed and they increase the pool of bugs with status closed. For bugs with status open the interesting part begins around May 2012 after the system is saturated/popular. It can be interpreted as a plot of how many bugs reside in the queue, the important part is that it is stable and does not show clear increasing trend.

The last plot shows arrival and departure rate of bugs from RFS queue, i.e. how many bugs are opened and closed each day.
